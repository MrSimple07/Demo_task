Task 1a

Bot Labels — SQL

Count unique LabelIDs per user for bot-related labels, created from September 2024 onwards.

SQL 
-- Task 1a: Bot labels per user, September 2024+
SELECT
    u.Name,
    COUNT(DISTINCT ul.LabelID) AS unique_label_count
FROM Users u
JOIN UserLabels ul
    ON u.ID = ul.UserID
WHERE
    LOWER(ul.LabelName) LIKE '%bot%'
    AND ul.CreatedAt >= '2024-09-01'
GROUP BY
    u.ID, u.Name
ORDER BY
    unique_label_count DESC;

Logic Breakdown

  • JOIN: inner join Users → UserLabels on UserID = ID
  • LOWER(LabelName) LIKE '%bot%': case-insensitive match — catches bot_vk, Bot_tg, botvk
  • CreatedAt >= '2024-09-01': September 2024 means >= the 1st; "not earlier than September" is inclusive
  • COUNT(DISTINCT LabelID): unique labels per user (prevents double-counting)
  • GROUP BY u.ID, u.Name: group by ID to avoid false merges if two users share the same name

Task 1b

Cumulative ARPU by Payment Month

For users who installed in January 2023, compute cumulative ARPU = running total payments ÷ cohort size.

SQL 
-- Task 1b: Cumulative ARPU — Jan 2023 install cohort
WITH jan_cohort AS (
    -- Step 1: isolate Jan 2023 installers
    SELECT
        user_id,
        installed_at
    FROM user
    WHERE
        installed_at >= '2023-01-01'
        AND installed_at < '2023-02-01'
),
cohort_size AS (
    -- Step 2: total cohort count (constant divisor)
    SELECT COUNT(DISTINCT user_id) AS n_users
    FROM jan_cohort
),
monthly_revenue AS (
    -- Step 3: aggregate payments by calendar month
    SELECT
        DATE_TRUNC('month', p.payment_at) AS payment_month,
        SUM(p.amount)                          AS monthly_revenue
    FROM payment p
    INNER JOIN jan_cohort jc
        ON p.user_id = jc.user_id
    GROUP BY 1
),
cumulative AS (
    -- Step 4: running sum over months
    SELECT
        payment_month,
        monthly_revenue,
        SUM(monthly_revenue) OVER (
            ORDER BY payment_month
            ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
        ) AS cumulative_revenue
    FROM monthly_revenue
)
SELECT
    c.payment_month,
    c.monthly_revenue,
    c.cumulative_revenue,
    ROUND(c.cumulative_revenue / cs.n_users, 2) AS cumulative_arpu
FROM cumulative c
CROSS JOIN cohort_size cs
ORDER BY c.payment_month;

Logic Breakdown

  • jan_cohort CTE: filters users where installed_at is strictly within January 2023 (avoids timezone/time bugs with < '2023-02-01')
  • cohort_size: computed once; denominator is total unique installers — never changes across months
  • monthly_revenue: aggregates payments from cohort users by calendar month using DATE_TRUNC
  • ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW: explicit window frame for the running sum
  • CROSS JOIN cohort_size: cleanly injects the constant divisor into the final calculation

Task 2

Data Mart Documentation
games_analytics — Marketing Intelligence Layer

1. Business Context

What is this mart and why does it exist?

The games_analytics mart serves the marketing team with a clean, pre-aggregated view of video game performance data from 1990–2010. The raw games.csv dataset contains inconsistencies (nulls, outliers, mixed-case strings) that make direct querying unreliable. This mart resolves them.

Business questions answered:

  • Which genres and platforms have the highest critical and user reception?
  • How did game releases trend over time, by platform?
  • What is the gap between critic and user scores by genre?
  • Which age ratings dominate which genres — and what does that mean for targeting?
  • Which platforms had the most prolific output in their peak years?

3. Transformation Logic

ETL Pipeline Steps

Step 1 — Ingest

Load raw games.csv into staging table stg_games_raw with all columns as TEXT to avoid parse failures.

Step 2 — Filter & Validate

Remove rows where: Year_of_Release is NULL or outside [1990, 2010]; Genre is NULL or empty; Critic_Score or User_Score is NULL; Rating is NULL. Approx. 15–20% of raw rows are excluded.

Step 3 — Normalize

TRIM and UPPER platform codes. TRIM and INITCAP genre names. Cast User_Score from string (handles 'tbd' → NULL). Cast Critic_Score to NUMERIC. Compute score_gap = (critic_score / 10.0) - user_score.

Step 4 — Load

Upsert into game_dim using name + platform + year_of_release as natural key. Rebuild genre_year_agg fully on each run (small table, full refresh is safe).

4. SQL DDL
SQL DDL 
CREATE SCHEMA IF NOT EXISTS games_analytics;

CREATE TABLE games_analytics.game_dim (
    game_id          SERIAL         PRIMARY KEY,
    name             VARCHAR(255)  NOT NULL,
    platform         VARCHAR(20)   NOT NULL,
    year_of_release  SMALLINT      NOT NULL CHECK (year_of_release BETWEEN 1990 AND 2010),
    genre            VARCHAR(50)   NOT NULL,
    critic_score     NUMERIC(5,1) NOT NULL CHECK (critic_score BETWEEN 0 AND 100),
    user_score       NUMERIC(4,2) NOT NULL CHECK (user_score BETWEEN 0 AND 10),
    rating           VARCHAR(5)    NOT NULL,
    score_gap        NUMERIC(5,2) GENERATED ALWAYS AS
                       ((critic_score / 10.0) - user_score) STORED,
    loaded_at        TIMESTAMP     DEFAULT NOW(),
    UNIQUE(name, platform, year_of_release)
);

CREATE TABLE games_analytics.platform_dim (
    platform_code  VARCHAR(20)  PRIMARY KEY,
    platform_name  VARCHAR(100),
    manufacturer   VARCHAR(100)
);

CREATE TABLE games_analytics.genre_year_agg (
    genre            VARCHAR(50),
    year_of_release  SMALLINT,
    game_count       INTEGER,
    avg_critic_score NUMERIC(5,1),
    avg_user_score   NUMERIC(4,2),
    avg_score_gap    NUMERIC(5,2),
    PRIMARY KEY (genre, year_of_release)
);

-- Indexes for common query patterns
CREATE INDEX idx_game_platform  ON games_analytics.game_dim (platform);
CREATE INDEX idx_game_genre     ON games_analytics.game_dim (genre);
CREATE INDEX idx_game_year      ON games_analytics.game_dim (year_of_release);
5. Example Queries

Q1 — Top genres by average critic score

SQL
SELECT genre, ROUND(AVG(critic_score), 1) AS avg_critic,
       COUNT(*) AS game_count
FROM   games_analytics.game_dim
GROUP BY genre
ORDER BY avg_critic DESC
LIMIT 10;

Q2 — Platform release count by year (top 5 platforms)

SQL
SELECT year_of_release, platform, COUNT(*) AS releases
FROM   games_analytics.game_dim
WHERE  platform IN (
    SELECT platform FROM games_analytics.game_dim
    GROUP BY platform ORDER BY COUNT(*) DESC LIMIT 5
)
GROUP BY year_of_release, platform
ORDER BY year_of_release, releases DESC;

Q3 — Score gap: critic vs user, by genre

SQL
SELECT genre,
    ROUND(AVG(critic_score / 10.0), 2) AS avg_critic_norm,
    ROUND(AVG(user_score),         2) AS avg_user,
    ROUND(AVG(score_gap),          2) AS avg_gap
FROM games_analytics.game_dim
GROUP BY genre
ORDER BY avg_gap DESC;

Q4 — Mature (M-rated) games by platform

SQL
SELECT platform, COUNT(*) AS m_rated_games,
    ROUND(AVG(critic_score), 1) AS avg_critic
FROM games_analytics.game_dim
WHERE rating = 'M'
GROUP BY platform
ORDER BY m_rated_games DESC;

Q5 — Year-over-year growth in game releases

SQL
SELECT
    year_of_release,
    COUNT(*)       AS releases,
    LAG(COUNT(*)) OVER (ORDER BY year_of_release) AS prev_year,
    ROUND((COUNT(*) - LAG(COUNT(*)) OVER (ORDER BY year_of_release))
        * 100.0 / NULLIF(LAG(COUNT(*)) OVER (ORDER BY year_of_release), 0),
    1) AS yoy_pct
FROM games_analytics.game_dim
GROUP BY year_of_release
ORDER BY year_of_release;
6. Limitations & Risks

⚠ User Score "tbd" values

Raw data contains the string "tbd" in User_Score. These must be treated as NULL during ingest or cast will fail. Approximately 5–8% of rows affected.

⚠ Platform name inconsistencies

Same platform may appear as "PS2" and "ps2" — UPPER() normalization required. Check for trailing spaces too (TRIM).

⚠ Duplicate game entries

Multi-platform releases appear as separate rows (same Name, different Platform). This is intentional design — but aggregate queries on Name must account for this to avoid inflated counts.

⚠ Score scale mismatch

Critic_Score is 0–100, User_Score is 0–10. Any cross-comparison (scatter plots, gap calculations) must normalize critic to 0–10 first.

⚠ Year_of_Release skew

Years after 2008 show declining counts — likely due to data collection lag, not actual industry decline. Interpret trend charts for 2008–2010 with caution.

Design Rationale

Why this visual direction

Dark theme with neon accents — the gaming industry context calls for a visual language familiar to the audience: high-contrast dark backgrounds with vivid accent colors evoke the aesthetic of game UIs and industry publications. This isn't decoration — it sets context and builds credibility with the intended user (marketing team in gaming).

Space Mono + Syne pairing — monospace for labels and data values (precise, technical, scannable) paired with Syne for headings (wide-set, editorial weight). The contrast between the two creates visual hierarchy without relying on size alone.

Three distinct accent colors — purple (#7c6eff), coral (#ff6e7c), and teal (#6effd4) are used consistently: purple for primary data, coral for warnings/comparisons, teal for positive/completion signals. This lets users read charts faster through color semantics.

Destroy-recreate chart pattern — rather than updating existing charts (which causes animation artifacts), each filter change destroys and rebuilds Chart.js instances. This keeps the dashboard visually clean and predictable.

Two-column chart grid with one wide slot — the stacked year/platform timeline needs horizontal space to be readable. Placing it full-width avoids squashing and gives it appropriate visual weight as the most complex chart.
