Task 1a

Bot Labels — SQL

Count unique LabelIDs per user for bot-related labels, created from September 2024 onwards.

SQL
-- Task 1a: Bot labels per user, September 2024+
SELECT
    u.Name,
    COUNT(DISTINCT ul.LabelID) AS unique_label_count
FROM Users u
JOIN UserLabels ul ON u.ID = ul.UserID
WHERE
    LOWER(ul.LabelName) LIKE '%bot%'
    AND ul.CreatedAt >= '2024-09-01'
GROUP BY u.ID, u.Name
ORDER BY unique_label_count DESC;

Logic Breakdown

  • JOIN: inner join Users → UserLabels on UserID = ID
  • LOWER(LabelName) LIKE '%bot%': case-insensitive match — catches bot_vk, Bot_tg, botvk
  • CreatedAt >= '2024-09-01': September 2024 inclusive, full date for strict SQL engine compatibility
  • COUNT(DISTINCT LabelID): unique labels per user (prevents double-counting)
  • GROUP BY u.ID, u.Name: group by ID to avoid false merges on duplicate names

Task 1b

Cumulative ARPU by Payment Month

For users who installed in January 2023, compute cumulative ARPU = running total payments ÷ cohort size.

SQL
-- Task 1b: Cumulative ARPU — Jan 2023 cohort
WITH jan_cohort AS (
    SELECT user_id FROM user
    WHERE installed_at >= '2023-01-01' AND installed_at < '2023-02-01'
),
cohort_size AS (
    SELECT COUNT(DISTINCT user_id) AS n_users FROM jan_cohort
),
monthly_revenue AS (
    SELECT
        DATE_TRUNC('month', p.payment_at) AS payment_month,
        SUM(p.amount) AS monthly_revenue
    FROM payment p
    INNER JOIN jan_cohort jc ON p.user_id = jc.user_id
    GROUP BY 1
),
cumulative AS (
    SELECT
        payment_month, monthly_revenue,
        SUM(monthly_revenue) OVER (
            ORDER BY payment_month
            ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
        ) AS cumulative_revenue
    FROM monthly_revenue
)
SELECT
    c.payment_month, c.monthly_revenue, c.cumulative_revenue,
    ROUND(c.cumulative_revenue / cs.n_users, 2) AS cumulative_arpu
FROM cumulative c
CROSS JOIN cohort_size cs
ORDER BY c.payment_month;

Logic Breakdown

  • jan_cohort: range filter >= '2023-01-01' AND < '2023-02-01' is index-friendly (sargable)
  • cohort_size: computed once with COUNT(DISTINCT) — constant denominator across all months
  • monthly_revenue: DATE_TRUNC groups payments into calendar months
  • ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW: explicit frame for running sum
  • CROSS JOIN cohort_size: injects the constant divisor cleanly without subqueries

Task 2

Data Mart Documentation
games_analytics — Marketing Intelligence Layer

1. Business Context

The games_analytics mart serves the marketing team with a clean, pre-aggregated view of video game data 1990–2010. Raw games.csv has inconsistencies (nulls, mixed-case strings, "tbd" values) — this mart resolves them.

Business questions answered:

  • Which genres and platforms have the highest critic and user reception?
  • How did game releases trend over time, by platform?
  • What is the gap between critic and user scores by genre?
  • Which age ratings dominate which genres?

3. ETL Pipeline
Step 1 — Ingest

Load raw games.csv into staging stg_games_raw with all TEXT columns to avoid parse failures.

Step 2 — Filter & Validate

Remove rows where Year_of_Release is NULL/outside [1990,2010], Genre is NULL/empty, Critic_Score or User_Score is NULL, Rating is NULL. ~15–20% of rows excluded.

Step 3 — Normalize

TRIM+UPPER platform codes. TRIM+INITCAP genres. Cast User_Score ('tbd' → NULL). Compute score_gap = (critic_score/10.0) - user_score.

Step 4 — Load

Upsert into game_dim using (name, platform, year_of_release) as natural key. Full refresh genre_year_agg each run.

4. DDL
SQL DDL
CREATE TABLE games_analytics.game_dim (
    game_id          SERIAL PRIMARY KEY,
    name             VARCHAR(255) NOT NULL,
    platform         VARCHAR(20)  NOT NULL,
    year_of_release  SMALLINT     NOT NULL CHECK (year_of_release BETWEEN 1990 AND 2010),
    genre            VARCHAR(50)  NOT NULL,
    critic_score     NUMERIC(5,1) NOT NULL CHECK (critic_score BETWEEN 0 AND 100),
    user_score       NUMERIC(4,2) NOT NULL CHECK (user_score BETWEEN 0 AND 10),
    rating           VARCHAR(5)   NOT NULL,
    score_gap        NUMERIC(5,2) GENERATED ALWAYS AS ((critic_score/10.0) - user_score) STORED,
    loaded_at        TIMESTAMP DEFAULT NOW(),
    UNIQUE(name, platform, year_of_release)
);
CREATE INDEX idx_game_platform ON games_analytics.game_dim (platform);
CREATE INDEX idx_game_genre    ON games_analytics.game_dim (genre);
CREATE INDEX idx_game_year     ON games_analytics.game_dim (year_of_release);
5. Example Queries

Q1 — Top genres by avg critic score

SQL
SELECT genre, ROUND(AVG(critic_score),1) AS avg_critic, COUNT(*) AS n
FROM games_analytics.game_dim GROUP BY genre ORDER BY avg_critic DESC LIMIT 10;

Q2 — Score gap by genre

SQL
SELECT genre,
    ROUND(AVG(critic_score/10.0),2) AS avg_critic_norm,
    ROUND(AVG(user_score),2)         AS avg_user,
    ROUND(AVG(score_gap),2)          AS avg_gap
FROM games_analytics.game_dim GROUP BY genre ORDER BY avg_gap DESC;

Q3 — YoY release growth

SQL
SELECT year_of_release, COUNT(*) AS releases,
    ROUND((COUNT(*) - LAG(COUNT(*)) OVER (ORDER BY year_of_release))
        * 100.0 / NULLIF(LAG(COUNT(*)) OVER (ORDER BY year_of_release),0), 1) AS yoy_pct
FROM games_analytics.game_dim GROUP BY year_of_release ORDER BY year_of_release;
6. Risks

⚠ User Score "tbd" values

~5–8% of rows have "tbd" string — must be cast to NULL during ingest or the pipeline fails.

⚠ Platform case inconsistencies

"PS2" vs "ps2" — UPPER()+TRIM() normalization required in ETL.

⚠ Score scale mismatch

Critic_Score is 0–100, User_Score is 0–10. Never compare directly — always normalize critic to /10 first.

⚠ Multi-platform duplicates

Same game on multiple platforms = multiple rows. Aggregate by (name, platform) not just name.

⚠ Year skew after 2008

Declining counts post-2008 likely reflect data collection lag, not industry decline.

Design Rationale

Why this visual direction

Dark theme with neon accents — high-contrast gaming aesthetic familiar to the target audience.

Horizontal genre bar — genre names are long strings. Horizontal bars give each label full width and are easier to scan top-to-bottom by count than rotated vertical labels.

Scatter plot removed — 836 points across 13 genre series created an unreadable cloud. The platform bar conveys a cleaner, more actionable insight at a fraction of the render cost.

80ms debounce — year sliders fire on every pixel drag. Debounce prevents unnecessary redraws mid-drag while keeping the dashboard responsive after release.
